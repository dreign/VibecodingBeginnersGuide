### 〇、格式与文件管理约定

#### 0.1 源文件格式

全书所有内容使用 **Markdown** 格式编写。

- 文件扩展名：`.md`
- 编码：UTF-8 without BOM
- 换行符：LF（`\n`）

**命名规则：**
- 文件夹：`章节号-中文章名`，如 `01-时代背景`。
- 文件：`章节号-小节号-主题词.md`，如 `01-01-从信息化到智能化.md`。
- 主题词用连字符 `-` 连接，保留中文，便于直接识别。

**交叉引用写法：**
由于文件分散在文件夹中，引用其他小节时使用相对路径：

```markdown
详见第1章第3节：[软件正在成为第二语言](../01-时代背景/01-03-软件正在成为第二语言.md)
```

引用图片时（`images/` 在根目录），从任一文件夹内的文件引用需向上返回两层到根目录：

```markdown
![图5-1：一人公司全生命周期工具链地图](../images/ch05-toolchain-map.png)
```

为了减少出错，可在每个文件夹内放置一个符号链接指向 `../images`（仅本地使用，不进入版本控制）。

**版本与日期标注：**
每个 `.md` 文件开头使用 HTML 注释记录最后更新日期及验证环境：

```markdown
<!--
  本节最后更新：2026-05-11
  验证环境：Node.js v22.x, Claude Code v1.x, 现代浏览器（Chrome/Firefox/Edge）
-->
```

### 一、写作红线：绝对禁区与必须遵守的原则

#### 1.1 技术承诺红线

| ❌ 绝对不能写 | ✅ 应该这样写 |
|--------------|--------------|
| “AI 能 100% 准确生成代码” | “AI 生成的代码需要通过运行来验证，这是 Vibe Coding 的核心循环” |
| “有了 AI，你永远不用学编程” | “AI 降低了启动门槛，但判断力、设计决策和调试思维仍是你的必修课” |
| “这个工具完美无缺，适合所有场景” | “这个工具在 X 场景下表现出色，但在 Y 场景下你可能需要考虑 Z” |
| “只需一句话就能做出完整产品” | “一个可用的产品需要多轮对话迭代，每一轮都在逼近你的真实意图” |

#### 1.2 数据与事实红线

- **所有性能数据必须标注来源与测试条件**。例如：不能写“生成速度提升 10 倍”，而应写“在 X 基准测试中，Y 模型在 Z 任务上比 W 模型快 X 倍（数据来源：XXX，测试日期：YYYY-MM-DD）”。
- **工具版本必须注明写作时的最新稳定版**，并在前言声明“工具更新快的本质”让读者有预期。
- **价格信息必须注明查询时间**，并说明“以官网最新定价为准”。
- **任何对比评测**（如模型 A vs 模型 B）必须写明：任务类型、测试数据、评判标准、测试日期。不搞“印象流”评测。
- **不引用未公开的内部数据或传闻**，所有引用的研究报告、博客文章均需提供可检索的出处。

#### 1.3 安全与法律红线

- 绝不教授如何绕过付费墙、破解软件或违反服务条款。
- 涉及 API 密钥、环境变量等安全实践时，必须强调“不要将密钥提交到 Git 仓库”。
- 引用开源代码时注明许可证类型，教会读者“在使用 AI 生成的代码前考虑版权风险”。
- 对“AI 生成的代码是否受版权保护”这一问题，如实说明“目前各国法律尚无定论”，不做确定性承诺。

#### 1.4 内容边界红线

- 本书聚焦“用 AI 构建软件”，不扩展到 AI 绘画、AI 视频、AI 音乐等领域（可在前言说明这是 Vibe 生态的一部分，但本书不展开）。
- 不深入大模型的数学原理（反向传播、注意力公式等），但提供“如果感兴趣可以去读 X”的延伸指引。
- 不推荐任何违反中国法律法规的工具或服务，如有提及仅供技术学习参考。

### 二、语言风格：友好、精确、有温度

#### 2.1 总体定位：一个经验丰富的搭档在你旁边说话

想象这样的场景：你和一位技术过硬、但毫无架子的朋友坐在咖啡馆。他打开笔记本电脑，一边敲代码一边跟你聊天，偶尔开个玩笑，但你问的任何问题他都会认真解释，不用大词吓你，也不假装简单。

**这就是本书的文字质感。**

#### 2.2 具体风格指南

**（1）用“你”和“我”，建立对话感**

- ✅ “你会看到终端里跳出一行行代码，就像有个隐形的搭档在帮你打字。”
- ❌ “用户将观察到终端输出，这表明 AI 正在执行代码生成任务。”

**（2）类比先行，术语后置**

- ✅ “Token 就像手机流量的最小计费单位——你每说一句话，都在消耗它。不同之处在于，AI 的回答也算在你的套餐里。”
- ❌ “Token 是语言模型处理文本的最小语义单元，其消耗量与上下文窗口大小呈正相关。”

**（3）允许口语化表达，但概念必须精确**

- ✅ “这个 Bug 看起来莫名其妙对吧？别急，我们把它贴回终端，让 AI 自己看看它干了什么好事。”
- ❌ “该程序错误的原因尚不明确，建议将错误信息重新提供给模型进行分析。”（意思对，但读着像实验报告）

**（4）用短句，拆分长逻辑**

- ✅ “模型会忘事。它的上下文窗口有限。对话越长，它能记住的就越少。所以，当你发现 AI 开始答非所问，可能不是它笨，是你的对话太长了。”
- ❌ “由于大语言模型的上下文窗口容量限制，当对话历史积累的 Token 数量超出其最大容量时，模型将表现出遗忘早期信息的现象，这并非能力退化而是架构限制。”

**（5）可以幽默，但不滑稽**

- ✅ “如果你把整个项目代码不加选择地全贴给 AI，那感觉就像问一个刚认识的人‘帮我修修我的生活’，对方大概率会懵。”
- ❌ “猛贴代码的样子真的很像信息轰炸，AI 都被你整不会了哈哈哈。”（过于随意，丢失了科教感）

**（6）不确定性要坦诚，但不制造焦虑**

- ✅ “这个问题目前没有一个确定的答案。但我们知道哪些路走不通，哪些方向值得跟踪。我会在书中标注这些‘已知的未知’。”
- ❌ “这块现在谁也不知道，就别管了。”

### 三、配图策略：用图像解释抽象

#### 3.1 配图原则

- **不追求装饰性配图**，每张图都要解释一个文字难以说清的概念。
- **复杂概念用“图解 + 标注”**，流程用“时间线”，对比用“并排图”。
- **所有配图预留 1-2 句说明文字的位置**，这句话要能独立讲清图的核心信息（防止只看图不看文的读者）。
- **配图风格**：清晰线框图 + 少量强调色，避免过度设计。技术概念图不追求艺术感，追求“一眼就懂”。

#### 3.2 全书建议配图位置与类型

| 章节 | 建议配图位置 | 图片类型 | 核心要传达的信息 |
|------|-------------|----------|------------------|
| 03-04 | 第 3 章结尾 | 阶梯图：四次编程范式解放，横轴=时间，纵轴=可使用人数 | 每次抽象跃升都释放了一大批创造者 |
| 04-04 | 第 4 章中段 | 饼图/环形图：传统开发周期中人类精力的分配（编码只占 20%） | 编码不是最大成本，沟通和等待才是 |
| 05-04 | 第 5 章核心 | 全景工具链流程图：一人公司从想法到收款的 8 个阶段 + 每个阶段的工具图标 | 一张图把“一人公司怎么跑通”说清楚 |
| 06-03 | 第 6 章中段 | Token 上下文窗口消耗示意图：一条对话时间线，上方显示 Token 累计量，标出窗口上限 | 对话越长≠越好，上下文会“满” |
| 07-02 | 第 7 章开头 | 四要素关系图：意图表达→上下文管理→AI 执行→即时反馈（循环箭头） | Vibe Coding 是一个闭环，不是单向指令 |

### 四、数据引用规范

#### 4.1 模型性能数据
- 必须注明：模型名称、版本、测试基准（如 HumanEval、SWE-bench）、测试日期、数据来源链接。
- 不能笼统说“Claude 比 GPT 强”，而要说“在 SWE-bench Verified 基准上，Claude 3.5 Sonnet 的得分为 X%（2024年X月测试），GPT-4o 的得分为 Y%（同期）”。

#### 4.2 用户数据与案例数据
- 如引用他人项目的 Star 数、下载量等，注明查询日期。
- 如使用“某独立开发者月收入 X 美元”的案例，注明来源（公开采访、社交媒体、社区帖子），并加注“收入受多种因素影响，不代表典型结果”。

#### 4.3 价格数据
- AI 模型 API 调用价格注明查询日期和对应模型版本。
- 工具订阅价格注明层级（免费/Pro/Team）和查询日期。

### 五、工具与代码示例规范

#### 5.1 工具名称
- 首次出现使用全称，括号注明常用缩写，如“Claude Code（CLI 终端工具）”。
- 尊重各工具的官方命名规范（如“GitHub Copilot”不写成“Github copilot”、“shadcn/ui”不写成“shadcn UI”）。

#### 5.2 代码片段
- 所有代码片段需在写作时实际运行验证，不写“伪代码”除非明确标注。
- 代码使用语法高亮格式，关键行添加注释。
- 不完整的代码片段用 `// ... 省略 ...` 明确标注。

#### 5.3 版本声明
每章文件夹中单独放置一个 `version.txt` 或在小节文件头部用注释统一定义该章运行时环境。建议每个小节头部注释包含验证环境。

### 六、章节语言风格差异化

| 部分 | 内容属性 | 语言调整 |
|------|---------|---------|
| 第一部分（认知） | 建立观念与愿景 | 故事感最强，类比密度高，允许宏观叙事 |
| 第二部分（概念） | 定义核心概念 | 类比依然重要，但概念精确度开始收紧 |
